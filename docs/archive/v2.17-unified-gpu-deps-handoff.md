## 0. TL;DR — what and why

The app has a **recurring fresh-install dependency failure problem** (numpy 2.x
breaking TensorFlow/OpenCV, missing scipy/absl/mediapipe, repair loops). The
**root cause is architectural**: the project has ~30 launchers + install sites
that install **divergent dependency subsets**, so a "self-heal / repair" layer
exists to patch the partial venvs after the fact — and that layer is brittle and
keeps needing fixes.

**The v2.17 goal (user mandate 2026-06-02, verbatim intent):**
> "what deps are needed for the entire app to work flawlessly, even in edge
> cases … and then we dont have to do any repairs and all this bullshit?"

So: **ONE canonical, GPU/OS-aware dependency bootstrap routine that every
launcher calls, installs the COMPLETE correct set up front, and verifies it —
so nothing is ever "missing" and repair becomes a rare verification, not an
architecture.**

The hard new requirement that a naive `pip install -r requirements.txt` gets
**wrong** (user mandate, confirmed against pytorch.org/get-started/locally):
- **Windows + working NVIDIA driver/CUDA-capable GPU** → install CUDA-capable
  torch (and CuPy if used).
- **Windows, no NVIDIA / broken / unsupported** → CPU torch, no CUDA.
- **macOS** → mac-appropriate torch (MPS/CPU), **never CUDA**.
- Do **NOT** install CUDA blindly. Do **NOT** assume CUDA Toolkit is present —
  detect the driver/GPU and **verify with a runtime probe**
  (`import torch; torch.cuda.is_available()` on Windows NVIDIA; the real CuPy GPU
  op for CuPy).

User confidence levels: separate-PR **10/10**, Windows NVIDIA-vs-CPU selection
needs explicit detection+probe **9/10**, macOS-must-not-attempt-CUDA **10/10**.

## 1. CRITICAL: a mature GPU subsystem ALREADY EXISTS — extend it, don't reinvent

This is the single most important fact for whoever does v2.17. **Most of the
GPU-aware machinery is already built and bot-reviewed.** Your job is largely to
**unify the launchers around it** and **close the install-divergence gaps**, not
to write GPU detection from scratch.

### Already-built GPU pieces (file:line):
- **`scripts/gpu_bootstrap.py`** (621 lines) — the primary GPU subsystem:
  - `detect_nvidia()` (L223-261): runs `nvidia-smi`, parses `Driver Version:` +
    `CUDA Version: N.N`.
  - `_resolve_nvidia_smi()` (L189-220): PATH + Windows fallbacks
    (System32, `NVIDIA Corporation\NVSMI`).
  - `_CUDA_TO_CUPY` map (L100-103): `{12: "cupy-cuda12x[ctk]", 13: "cupy-cuda13x[ctk]"}`.
  - `install_cupy()` (L304-351), `probe_cupy()` (L264-301): real GPU-op probe.
  - `bootstrap()` (L414-582): stamp-cached state machine →
    `.launcher_state/gpu_status.json` (states: `gpu_ready`/`no_nvidia`/
    `install_failed`/`cached_no_nvidia`). Opt-out `KLING_SKIP_GPU_BOOTSTRAP=1`.
    Lock `gpu_bootstrap.lock`. Always exits 0.
  - Already wired into `launchers/windows/run_gui.bat` (L364-366,
    `--quiet-if-cached`) and `run_gui.sh` (L154-156).
- **`dependency_health_check.py`** — torch CUDA-failure detection + CPU fallback:
  - `_TORCH_CUDA_FAILURE_SIGNATURES` (L156-172): cudart/cudnn/cublas/cusolver/
    cufft/nvrtc/nccl etc.
  - `_TORCH_CPU_INDEX_URL = "https://download.pytorch.org/whl/cpu"` (L176).
  - `check_runtime_dependencies()` (L201-357): probes `torch.cuda.is_available()`
    (L309) + build-vs-runtime mismatch via `torch.version.cuda` (L316-328).
  - `run_torch_cpu_fallback()` (L425-472): force-reinstalls torch from the CPU
    index, pinned to the installed public version (strips `+cuXXX`).
  - `run_repair()` (L475-542): CPU fallback first if CUDA break, then face-stack.
  - `_is_cuda_only_failure()` (L589-596) → exit 0 + WARN when only CUDA broke
    (app still launchable on CPU).
- **rPPG CuPy GPU path:** `rPPG/rppg_injector.py` (L87-197) optional
  `import cupy` + `cupyx.scipy.ndimage`, CPU fallback on import failure.

**Production code does NOT call `torch.cuda.*`** (verified — only test fixtures
do). torch is used by DeepFace's anti-spoofing classifier; CUDA only matters for
speed, not correctness. So "CPU fallback is always safe" holds.

### What's MISSING / divergent (the actual v2.17 work):
The GPU bootstrap handles CuPy + torch-CUDA-failure-recovery, but **torch itself
is installed as a plain `torch>=2.2,<3` from requirements.txt with NO hardware
selection at install time** — the CUDA build is only reached via the *repair*
path after a failure. v2.17 should make torch hardware-SELECTED **at install
time** (NVIDIA → CUDA wheel index; else CPU index; mac → default), folding into
the existing gpu_bootstrap + health-check logic rather than duplicating it.

## 4. Stamps, locks, venv layout (preserve these invariants)

**Venvs:** Windows canonical = `venv\` (resolver also probes `.venv311`, `.venv`).
macOS canonical = `.venv-macos\` (dev venv `.venv311`). Sub-projects SHARE the
root venv by default (rPPG/similarity/resemble/oldcam all prefer `REPO_ROOT/venv`).

**Stamps** (`.launcher_state/`):
- Windows: `deps_<datesize-of-reqs+oldcam-reqs>.ok` — **HEALTH_OK-gated**
  (run_gui.bat L345-352). `run_cli.bat` writes it **unconditionally** (BUG to fix).
- macOS: `.venv-macos/.requirements.sha256` (sha of reqs+constraints) +
  `.health.sha256` (refreshed only on success). `run_cli.sh` SKIPS health when
  the two stamps match (BUG to fix — once matched, later breakage never re-detected).
- Per-subproject: `oldcam_v*_*.ok`, `similarity_gui_*.ok`, `resemble_gui_*.ok`,
  `auto_*.ok` — each gated on its own minimal import re-probe.
- **v2.17 stamp design:** stamp key MUST include reqs + constraints + INSTALLER
  VERSION + platform/GPU mode, so a GPU-mode change or installer bump invalidates.

**Locks** (mkdir-based, `.launcher_state/`): `setup.lock` (bootstrap; win
forfiles-1day stale, mac `LOCK_STALE_SECONDS=600`), `gpu_bootstrap.lock`
(`LOCK_STALE_SECONDS = PIP_INSTALL_TIMEOUT + 300 = 1200s`, deliberately > pip
timeout), `rppg_setup.lock` (bounded retry counter, force-break ~30 min). The
rule (from gpu_bootstrap): **LOCK_STALE_SECONDS must exceed the pip install
timeout** so a live install isn't force-broken — preserve in any new lock.

**Python resolver:** `scripts/win_resolve_python.bat` (296 lines) — CALLed
(no setlocal), sets `VENV_PYTHON`/`RESOLVED_PYTHON`/`RESOLVE_RC`. Order: existing
venvs → `SELFIEGEN_PYTHON` override → `py -3.11/3.12/3.10/3.9` → `python` PATH →
common dirs → auto-install Python 3.12 (winget then python.org). Version gate
3.9-3.12. Reuse it; don't duplicate.

## 6. v2.17 proposed phases (suggested; refine in plan mode)

**Phase 0 — branch + read.** Branch `fix/unified-gpu-aware-dependency-bootstrap`
off fresh `main`. Read this doc's linked references (§9). Confirm #67 is merged.

**Phase 1 — design the canonical installer contract.** One routine per platform
(Windows + macOS) that: (a) resolves python (reuse win_resolve_python.bat),
(b) detects GPU (reuse gpu_bootstrap.detect_nvidia), (c) installs the FULL
requirements.txt with `-c constraints.txt` + mediapipe `--no-deps`, (d) selects
torch by hardware (NVIDIA→CUDA index, else CPU index, mac→default), (e) verifies
via runtime probe, (f) writes a stamp keyed on reqs+constraints+installer-version+
GPU-mode. Probably a Python module (`scripts/install_deps.py` or extend
gpu_bootstrap) the .bat/.command launchers all call, so the logic lives in ONE
testable place, not 30 shell scripts.

**Phase 2 — full-set health probe.** Add scipy/absl/mediapipe to
`check_runtime_dependencies`; make REPAIR_PACKAGES + PYTHON_DEPENDENCIES the full
set. A partial venv must FAIL health.

**Phase 3 — converge the launchers.** oldcam/similarity/resemble stop their
divergent mini-installs — either call the shared routine or guarantee they run
only against an already-complete shared venv. Fix run_cli.bat (unconditional
stamp) + run_cli.sh (skip-health-when-matched). Add `-c constraints` to
build_gui_exe.bat. Fix run_auto.bat + build_gui_exe.bat mediapipe `--no-deps`.

**Phase 4 — torch GPU selection at install time.** NVIDIA detected → install
torch from the CUDA wheel index matching the driver's CUDA version; else CPU
index; mac → default. Verify `torch.cuda.is_available()` on Windows-NVIDIA.

**Phase 5 — tests + fresh-zip smoke (MANDATORY gate).** See §7.

## 10. Non-negotiable guardrails (carry from PR #67 process)

- **`.bat`/`.cmd`: write via PowerShell `WriteAllText` with explicit `` `r`n ``,
  NEVER the Write/Edit tool** (they emit LF + garble batch). Chain
  `python-write && git add` in ONE Bash call (the `/dev/null` linter rewrites
  `>nul`→`/dev/null` the instant the file sits unstaged).
- **After every Edit/Write on a tracked file: `git diff --stat` before commit.**
  A whole-file change for a small edit = EOL flip; restore via byte-level rewrite
  matching the committed EOL (`git ls-files --eol`). Some files are mixed-EOL.
- **Never `git add -A`/`git add .`** — `build_gui_exe.bat` autocrlf-flips on this
  box and must never enter a commit; stage per-file.
- **No `nul` files** (`find . -name nul -type f -delete`; phantom POSIX matches of
  the Windows NUL device are harmless — confirm with PowerShell it's not real).
- **Thread `-c constraints.txt` through every NEW pip site** + add it to the
  launcher guard test.
- **Autonomous PR loop:** branch → commit → push → trigger bots (@coderabbitai
  @codex @gemini-code-assist @sourcery-ai) + spawn code-reviewer subagent on the
  FULL `git diff main...HEAD` in parallel → fix everything reasonable → loop →
  STOP at merge gate (user merges). The personal-zip builder is the uncommitted
  `distribution/build_release_personal.py`; bump `app_version.py` BEFORE building.
- **Don't defer fixable findings** ("pre-existing"/"V2" excuses are a flagged lazy
  pattern). Fix now unless it genuinely risks regression or needs a product call.
