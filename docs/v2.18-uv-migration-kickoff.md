# v2.18 Kickoff — Full `uv` Migration (with lockfile)

> **HOW TO USE THIS FILE:** Paste the whole thing as the first message in a
> **new** Claude Code chat in this repo. It is the complete, self-contained
> brief for migrating the project's dependency management to
> [`uv`](https://github.com/astral-sh/uv) with a committed `uv.lock`. It folds
> in the hard-won lessons from the v2.10–v2.17 dependency saga so the migration
> doesn't reopen any of those holes.

---

## Mission

Migrate **all** dependency management in this repo from the current
`pip` + `requirements.txt` + `constraints.txt` + hand-rolled
stamp/health/launcher machinery to **`uv` with a committed `uv.lock`**, such
that a fresh end-user install is **faster, reproducible, and reliable** on
**both Windows and macOS**, with **GPU acceleration auto-enabled when an NVIDIA
GPU is present** — and **without breaking anything that currently works**.

The end user is **non-technical** and installs by double-clicking a launcher
(or unzipping a release zip and double-clicking). They never see a terminal
command. That constraint drives every decision below.

This is a **big job** and a **separate PR** on a **fresh branch**
(`feat/uv-migration` or similar). It must NOT touch `main` until the user
explicitly approves the merge.

## Non-negotiable working agreement (read first)

1. **Proper TDD.** Write the test FIRST, watch it fail, then implement. For
   dependency behaviour the test must **REAL-IMPORT the deep symbol** (e.g.
   `from mediapipe.tasks.python.vision import FaceLandmarker`,
   `import scipy`, `torch.cuda.is_available()`), **not** grep source text.
   Source-text "OK" has shipped broken venvs here before. (See the project
   memory `feedback_tdd_real_import_probe_not_just_text`.)
2. **Be fully autonomous.** Run the branch → commit → push → trigger bots +
   spawn a `code-reviewer` subagent on the full branch diff → fix → loop →
   hold-at-merge-gate workflow **end-to-end without asking permission** for any
   intermediate step (opening the PR, committing, engaging bots, applying
   fixes). The ONLY thing requiring explicit user approval is the **final
   merge**. The full loop is codified in this repo's `CLAUDE.md` ("Default
   Workflow After Feature Work"). Follow it.
3. **Don't break what works.** The app currently launches and runs on both OSes
   from a fresh install (v2.17, `main`). Every existing test must stay green.
   Migrate behind a flag / in phases (see "Safety net" below) so a uv
   regression can't brick all ~30 launchers at once. Rollback = **this branch
   simply never merges** (it's a fresh branch off `main`); make sure that's
   literally true — no changes that take effect before merge, no shared-state
   migrations that outlive the branch.
4. **Use the MCP tooling proactively** — `serena` for symbol lookup/edits,
   `context7` / `ref-tools` for `uv` docs, `exa` for current `uv` + PyTorch-on-uv
   practices, `desktop-commander` for process/fs control. (Global mandate in
   `~/.claude/CLAUDE.md`.)
5. **Windows + macOS parity is the whole point.** Every change must be verified
   (or explicitly flagged as un-verifiable) on the OS you're not on. The user
   bounces between macOS (primary dev) and Windows (verification). See the
   `docs/cross-os-bounce-traps.md` pre-PR matrix.

## Why uv, and the one thing that makes it non-trivial here

`uv` gives us: a single fast resolver, a real lockfile (`uv.lock`) with hashes,
reproducible `uv sync`, and a managed Python — which would replace the bespoke
`win_resolve_python.bat` auto-install dance. **Once deps are installed the
runtime is identical**, so the win is install speed + reproducibility +
deleting a lot of fragile shell/stamp code.

**The hard part — GPU/OS-aware torch (and CuPy) selection under uv.** A naive
`uv sync` resolves CPU torch from PyPI. This project needs:

- **Windows + working NVIDIA** → CUDA torch wheel (from the PyTorch CUDA index)
  **and** CuPy (`cupy-cudaNNx[ctk]`), with a **runtime probe**
  (`torch.cuda.is_available()`) and **CPU fallback** if the CUDA build is
  runtime-broken.
- **Windows, no/broken NVIDIA** → CPU torch.
- **macOS** → default (MPS/CPU), **never CUDA**.

The torch/CUDA index selection, the **CuPy↔numpy<2 conflict**, and the
**nvrtc DLL discovery** problem (all detailed below) are exactly where a uv
migration is most likely to regress. The v2.17 work already solved all three
under pip — your job is to reproduce that correctness under uv, ideally more
declaratively. Investigate `uv`'s mechanisms for this **before** designing:
- `[tool.uv.sources]` / `[tool.uv.index]` with `explicit = true` for the
  PyTorch CUDA index (e.g. `https://download.pytorch.org/whl/cu128`).
- Platform/marker-conditional sources, and whether uv can express
  "CUDA index on win+nvidia, CPU elsewhere" declaratively or whether a
  post-`uv sync` selector step (like today's `scripts/gpu_bootstrap.py
  --select-torch`) is still needed. **Verify against current `uv` docs — do
  not assume.**

## Hard constraints carried from the v2.10–v2.17 dependency saga (DO NOT reopen)

These caused real fresh-install breakages. The migration must preserve every
one of them in the new uv world (likely as `uv.lock` pins + markers):

1. **`numpy < 2`** everywhere. TensorFlow 2.16.2's `ml-dtypes~=0.3.1` needs
   numpy 1.26.x. An unconstrained resolve pulls numpy 2.x and breaks TF import
   (the recurring "numpy.core.umath failed to import" Face Crop crash).
2. **All three opencv variants `< 4.12`.** opencv 4.12+ declares `numpy>=2`,
   conflicting with the numpy<2 pin → `ResolutionImpossible`.
   (`opencv-python`, `opencv-python-headless`, `opencv-contrib-python`.)
3. **`tensorflow==2.16.2`** (+ `tensorflow-intel==2.16.2` on win32) pinned
   explicitly — deepface declares only a loose `tensorflow>=1.9.0`.
4. **`tf-keras==2.16.0`** pinned; the app sets `TF_USE_LEGACY_KERAS=1` (keras 3
   breaks deepface). Tests rely on `conftest` setting this at import time.
5. **`absl-py>=2.3,<3` pinned explicitly.** mediapipe is installed `--no-deps`
   today (so it can't pull numpy 2.x), which skips its `absl-py~=2.3`; the
   TF-side absl can fail to land on a fresh venv. rPPG's injector imports
   `absl.logging` (now guarded, but still wants it present).
6. **mediapipe is special.** Today it's installed `--no-deps` THEN its real
   runtime deps are installed separately (`matplotlib`, `opencv-contrib-python<4.12`,
   `sounddevice`, `numpy<2`). The killer bug: `mediapipe.tasks.python.vision`
   (the FaceLandmarker used by rPPG + oldcam) imports **matplotlib at module
   load** — a bare `import mediapipe` passes but the real Tasks-API import
   crashes with "No module named matplotlib", silently forcing `-NORPPG` on
   every run. **The health probe MUST validate the deep Tasks-API symbol, not
   `import mediapipe`.** Under uv, decide: can `uv.lock` carry mediapipe's full
   dep tree safely (numpy still pinned <2 by the lock), eliminating the
   `--no-deps` hack? If so that's a real simplification — but prove the lock
   keeps numpy<2 with mediapipe's deps resolved.
7. **`scipy>=1.11,<2`** is a real runtime dep (probed by the health check).
8. **`cupy-cudaNNx[ctk]>=13.6,<14`** — CuPy 14.x requires numpy>=2.0, which is
   mutually exclusive with the numpy<2 face stack. Must stay `<14`.
   (Memory: `project_cupy_numpy2_conflict`.)
9. **nvrtc DLL discovery on Windows:** CuPy `[ctk]` wheels drop CUDA DLLs under
   `site-packages/nvidia/*/bin`; Windows Python 3.8+ ignores PATH for
   extension-module deps, so `rPPG/rppg_injector.py` calls
   `os.add_dll_directory()` (handles retained in a module-level list so the GC
   doesn't drop them) + PATH prepend before `import cupy`. This is runtime code,
   unaffected by the package manager — but **don't delete it**, and re-verify it
   still works against uv-installed CuPy wheels.
10. **The hash-pinned lockfile must be generated on Windows**, not macOS — the
    current `requirements-hashed.txt` is pinned to Intel/Windows TensorFlow
    wheels; a macOS regen overwrites the Windows wheel set and breaks the
    bundled release. `uv.lock` is cross-platform by design (it locks per-marker),
    so confirm `uv lock` on either OS produces a lock that resolves correctly on
    **both** — this is a key thing to verify, not assume.

The authoritative current pin set lives in `requirements.txt` + `constraints.txt`
+ `dependency_health_check.py::REPAIR_PACKAGES`. Read all three.

## FOLD IN: the automation/batch CLI (just fixed in PR #69 — keep it working)

A sibling PR (`fix/automation-cli-v217-headless-batch`, PR #69) just fixed the
**fully-automated batch CLI** so it works with the v2.17 dependency bootstrap.
The uv migration MUST keep that path working — it's a first-class surface, not
an afterthought. Specifically:

- **Headless entry:** `kling_automation_ui.py --batch <root> [--limit N]
  [--reprocess MODE] [--yes]` runs the pipeline non-interactively (no stdin),
  exit codes `0`=clean / `1`=could-not-run / `2`=ran-but-failed-or-manual_review.
- **Launchers:** `run_auto.bat` (Windows) + `run_auto.command` (macOS) are
  **thin wrappers** that set `KLING_NONINTERACTIVE=1` (so the chain never
  `pause`/`read`s and hangs an unattended job) and delegate to the canonical
  CLI launcher chain with `--batch` injected. Under uv, the canonical chain
  becomes `uv sync` + `uv run` (or equivalent); `run_auto.*` must stay a thin
  wrapper over whatever the new canonical chain is, and the
  `KLING_NONINTERACTIVE` pause/read guards must survive.
- **Both interactive and batch** modes share one venv + one bootstrap. Don't
  let uv re-introduce divergence between the GUI launcher, the CLI launcher, and
  the batch launcher — the v2.17 work's whole thesis was "one canonical
  per-platform installer all launchers call." uv should make that EASIER (one
  `uv sync`), not harder.
- There are ~13 sub-launchers (oldcam-v7..v24, similarity standalone, resemble)
  that today run a **shared-venv preflight** and do NOT install divergent
  subsets. Under uv they should `uv run` against the same project env. Preserve
  that "no sub-launcher installs its own subset" invariant.

Tests to keep green: `tests/test_automation_headless_batch.py`,
`tests/test_run_auto_launcher_parity.py`, `tests/test_automation_*`,
`tests/test_launcher_*`, `tests/test_sublauncher_preflight.py`.

## DECISION REQUIRED (surface to the user early, in plan mode):
## keep the stamp/health layer, or go uv-native?

Today there's a hand-rolled layer: per-launcher `.ok` / `.health.sha256`
**stamps** (skip dep sync when unchanged) + a **runtime health probe**
(`dependency_health_check.py --mode check/repair`) that re-validates the venv
every launch and auto-repairs (numpy re-pulled, AV-quarantined DLL, partial
wheel). The user's framing: *"once deps are installed it kinda doesn't matter
since we got the stamping anyway."*

`uv sync` is itself fast + idempotent + lock-verified, which **subsumes the
stamp** (no need to fingerprint requirements files — the lock IS the
fingerprint). But `uv sync` checks the **lock**, not whether the installed
wheels still **import** (it won't catch an AV-quarantined TF DLL or a
numpy that some other process force-reinstalled). So:

- **Option A — uv-native, drop the stamp, keep a thin health probe.** Replace
  the stamp with `uv sync` (fast no-op when in sync); KEEP a lightweight
  runtime import-probe (the deep-symbol one) as the safety net for
  post-install corruption, with `uv sync` as the repair instead of
  `--force-reinstall`. **Recommended starting hypothesis** — simplest, keeps
  the one real safety net.
- **Option B — keep the stamp layer on top of uv.** More code, but a known
  quantity. Probably redundant with `uv sync`.

Bring a recommendation + the trade-off to the user in plan mode. Don't silently
pick one.

## Phased plan (refine in plan mode; this is a sketch, not a contract)

- **Phase 0 — Investigate + decide.** Read the v2.17 machinery end-to-end
  (`scripts/gpu_bootstrap.py`, `dependency_health_check.py`,
  `dependency_checker.py`, all launchers, `constraints.txt`, `requirements.txt`,
  `requirements-hashed.txt`). Verify current `uv` capabilities for
  CUDA-index/marker-conditional torch via context7/exa. Produce a
  `pyproject.toml` design + the stamp-vs-uv-native recommendation. **EnterPlanMode
  and get the user's sign-off before writing code.**
- **Phase 1 — `pyproject.toml` + `uv.lock`.** Author `pyproject.toml`
  (dependencies = the current pinned set, with markers), generate `uv.lock`,
  verify it resolves on **both** OSes (numpy<2 held, opencv<4.12, TF 2.16.2,
  tf-keras, absl, scipy, mediapipe deep import). Add a real-import smoke test.
- **Phase 2 — GPU torch + CuPy under uv.** Reproduce the v2.17 selection logic:
  declarative `[tool.uv.sources]`/index if uv supports it, else a thin
  post-sync selector. Runtime `torch.cuda.is_available()` probe + CPU fallback.
  CuPy `[ctk]` + the nvrtc DLL retention (verify against uv-installed wheels).
  Real GPU test on the NVIDIA box (the user has an RTX 4090; v2.17 measured
  ~4.1× full rPPG pipeline GPU vs CPU — confirm parity).
- **Phase 3 — Launchers call uv.** One canonical per-OS launcher path that does
  `uv sync` (+ torch selection) then `uv run`. Rewrite GUI + CLI + batch +
  sub-launcher chains to use it. Keep `KLING_NONINTERACTIVE` guards.
  Bootstrap `uv` itself when absent (winget/standalone installer on Windows;
  the official install script on macOS) — the end user has no terminal.
- **Phase 4 — Retire/repurpose the old machinery.** Per the stamp decision:
  delete the stamp code or keep a thin probe. Update `distribution/` (frozen
  build + release zip) so the shipped artifact uses uv too (or a uv-exported
  set — decide). Update `docs/` + `CLAUDE.md` dep sections.
- **Phase 5 — Full test matrix + fresh-zip smoke** (mandatory gate):
  Win-GPU, Win-CPU, macOS — each: extract a fresh zip, double-click launch,
  click Face Crop, run a `--batch` job. Record which paths were actually run
  vs simulated in the PR description (the user's standing requirement).

## Safety net (rollback)

- Fresh branch off `main`; **never commit to `main`**. Rollback = don't merge /
  delete the branch. Confirm nothing in the branch takes effect pre-merge
  (no machine-global installs that persist, no shared-state schema migration
  that an un-merged branch would leave behind).
- Phase the launcher cutover so a uv failure can fall back to the existing pip
  path during development (e.g. behind a `KLING_USE_UV=1` env flag until Phase 4
  flips the default). This lets you test uv on one machine without bricking the
  other.
- Keep `requirements.txt`/`constraints.txt` in the tree until the uv path is
  proven on both OSes; remove them only in the final phase.
